# AI_TOOLS 專案架構及功能分析（中文交接版）

- 分析路徑：`H:\AI\AI_TOOLS`
- 產出日期：2026-03-19
- 文件用途：提供 Python + Django 專案移交，協助新接手工程師快速上手。

## 1) 專案總覽（用途、技術棧、執行方式）
- 核心定位：多子系統入口網站，整合公文、RAG、翻譯、語音、PDF 與簡報生成。
- 後端框架：Django 6.0。
- 資料層：SQLite（Django 內部）+ Oracle / SQL Server / Sybase（由 DB Factory 統一接入）。
- LLM 層：`webapps/llm/llm_factory.py`，支援 AUTO / OPENAI / GOOGLE / OLLAMA。
- 向量檢索：ChromaDB（供 RAG 使用）。
- 啟動方式：`python manage.py runserver 8000`。

## 3) Django 專案 / 各 app 功能分析
- `portal`：入口首頁、whoami 診斷、使用紀錄查詢與 ACL 觀測。
- `doc`：公文模板管理、附件重點解析、草稿生成、Sybase 來文查詢。
- `meetingreply`：待辦清單、RAG 內容擷取、回覆草稿組裝。
- `rag_oracle`：RAG 問答 API，並提供同步 SQL Server 至 Chroma 指令。
- `translator` / `comment` / `student`：翻譯、評語與格式輸出。
- `tts` / `pdf` / `text2pptx` / `graph2doc` / `excelproc`：文件與多媒體處理子系統。

## 5) 主要資料模型與資料流
- 主要模型：
- `webapps/portal/models.py`：`PortalUsageLog`（使用紀錄、whoami_json、索引）。
- `webapps/doc/models.py`：`DocumentTemplate`（JSON schema、scope/owner、模板內容）。
- 典型資料流：
1. Request 進入 `webproj/urls.py`。
2. 經 `portal` middleware 處理 proxy/script_name 與登入身分。
3. 由 app view 呼叫 DB Factory / LLM Factory。
4. 回傳 API 或頁面結果。

## 10) 風險與技術債（按高/中/低）
- 高：缺少 `.env.example`，新環境參數遺漏風險高。
- 高：`DEBUG` / `DJANGO_DEBUG` 在 `.env` 有開啟值，正式環境需嚴管。
- 中：外部依賴多（DB/LLM/代理），需補齊健康檢查與 fallback。
- 中：測試偏腳本，回歸測試一致性不足。
- 低：requirements 存在多版本檔案，建議整併。
