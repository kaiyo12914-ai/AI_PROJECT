# doc 子系統精進策略方案

## 一、方案目標

針對 `webapps/doc` 子系統，建立「可維護、可測試、可擴充、可追蹤」的工程體質，並在不影響既有業務功能（公文產製、來文查詢、範例匯入、附件處理）的前提下，分階段完成技術債清理與架構升級。

---

## 二、現況摘要（重點問題）

### 1) 阻斷級問題
- `management/commands/fill_meta.py` 存在縮排錯誤（`IndentationError`），會造成語法檢查失敗。
- `management/commands/clean.json.py` 檔名不符合 Django command 探測慣例，命令不可用風險高。

### 2) 維護風險
- 重複檔案與多版本並存：
  - `docService.py`（根目錄）與 `services/docService.py`（正式服務層）並存。
  - `static/doc/incoming_sybase.js` 與 `static/doc/js/incoming_sybase.js` 並存且內容不同。
  - `rules/org_level_map.json` 與 `services/rules/org_level_map.json` 並存且版本不一致。
- 大型檔案過多（例如 `views_sybase_query.py`、`views_generate.py`、`views_parse.py`），理解與回歸成本高。

### 3) 品質工程不足
- 測試覆蓋率偏低（測試程式碼量遠低於主程式碼量）。
- `except Exception` 使用頻率高，容易吞掉根因與降低除錯效率。

---

## 三、精進總體策略

採「四階段、漸進式重構」：

1. **Phase A：止血與可發布**（先修阻斷）
2. **Phase B：一致性與單一真相來源（SSOT）**
3. **Phase C：模組化重構與可觀測性**
4. **Phase D：品質閘門與持續治理**

核心原則：
- **先穩定再重構**
- **每次改動都可回退**
- **每個階段皆有可量化驗收標準**

---

## 四、分階段執行方案

## Phase A（第 1 週）— 止血與可發布

### 目標
恢復基本可建置、可執行、可檢查狀態。

### 執行項目
1. 修正 `fill_meta.py` 縮排錯誤並通過語法檢查。
2. 將 `clean.json.py` 正式更名為 `clean_json.py`，並驗證可被 Django command 載入。
3. 建立最小 smoke check：
   - `python3 -m compileall webapps/doc`
   - 主要 URL 路由載入檢查
4. 針對「不可影響現行功能」的核心 API 建立人工驗證清單（查詢、匯入、下載）。

### 驗收標準
- compileall 0 error。
- 管理命令可正常列出與執行。
- 核心頁面與 API 無 500 錯誤。

---

## Phase B（第 2 週）— 一致性與 SSOT

### 目標
消除重複來源，降低「改到錯檔」風險。

### 執行項目
1. 決議並保留唯一服務實作入口（`services/docService.py`），其他同名檔轉為：
   - 移除，或
   - 明確標註 deprecated 並導向單一入口。
2. 統一 `incoming_sybase.js` 實體來源，移除未引用版本。
3. 規則檔 `org_level_map.json` 整併為單一來源，並建立載入順序規範。
4. 增加啟動時一致性檢查（版本號、必要欄位、mapping schema）。

### 驗收標準
- 關鍵模組皆為單一來源。
- 專案內不再存在功能性重複檔案（保留備份需轉移到 archive 區）。
- 規則檔版本一致、讀取邏輯一致。

---

## Phase C（第 3～4 週）— 模組化重構與可觀測性

### 目標
降低單檔複雜度，提升故障可診斷性。

### 執行項目
1. 將大型 view 檔拆分成以下層次：
   - `views/api/*.py`（輸入/輸出）
   - `services/*.py`（業務邏輯）
   - `repositories/*.py`（資料存取）
   - `parsers/*.py`（文件解析）
2. 將泛抓例外逐步替換為明確例外類型，至少覆蓋高頻熱點。
3. 導入結構化日誌欄位：
   - `request_id`, `login_user`, `plant`, `grsno`, `endpoint`, `error_code`
4. 將硬編碼設定（如允許帳號）移至 settings/.env。

### 驗收標準
- 三大大型檔案完成切模組，單檔行數明顯下降。
- 錯誤日誌可追蹤到具體請求與參數。
- 主要權限配置已外部化。

---

## Phase D（第 5 週起）— 品質閘門與持續治理

### 目標
把精進成果制度化，避免回彈。

### 執行項目
1. 建立 doc 子系統 CI 最低門檻：
   - compileall
   - lint（如 ruff）
   - unit tests
2. 增補測試優先順序：
   - DB routing
   - 權限判斷
   - query/import/blob 主要流程
   - 規則檔解析與 fallback
3. 設定 Code Review 檢核項：
   - 禁止新增無理由重複檔
   - 禁止新增未說明之 `except Exception`
   - 新增功能需附最小測試
4. 每月進行一次「技術債盤點 + 迭代計畫回顧」。

### 驗收標準
- 主要 PR 均需通過 CI。
- 每月可量化改善（錯誤率、平均修復時間、測試覆蓋增量）。

---

## 五、治理機制（建議）

### 1) 角色分工
- **架構責任人**：審核模組邊界、命名與依賴方向。
- **品質責任人**：維護測試策略與 CI 門檻。
- **業務責任人**：確認公文流程行為一致。

### 2) 變更管控
- 每個 Phase 皆以「小批次 PR」推進。
- 每次合併都附：影響範圍、回退方式、驗證紀錄。

### 3) KPI 建議
- P0 缺陷修復 SLA（24h 內）。
- 500 錯誤率月降幅。
- 平均定位時間（MTTD）與平均修復時間（MTTR）。
- doc 子系統測試覆蓋成長率。

---

## 六、風險與對策

1. **重構期間功能回歸**
   - 對策：先加 smoke tests + 核心流程腳本驗證。
2. **舊檔依賴不明導致刪除風險**
   - 對策：先全域依賴掃描，再 deprecate 一個版本周期後移除。
3. **人力被日常需求擠壓**
   - 對策：每週固定保留 20% 工時處理技術債。

---

## 七、預期成果

完成本方案後，doc 子系統可達成：
- 發版前可自動化檢查（避免低階語法錯誤上線）
- 架構邊界清楚（降低維護人員接手門檻）
- 問題可追蹤（縮短除錯時間）
- 品質可持續提升（以 CI 與規範防止退化）

---

## 八、執行建議（下一步）

建議本週先啟動 **Phase A + Phase B 的前半段**：
1. 修語法錯誤與命令檔命名。
2. 完成重複檔案盤點與保留策略表。
3. 同步建立第一版 smoke check（可先手動，後續上 CI）。
