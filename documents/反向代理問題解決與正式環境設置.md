# 問題
- 靜態檔案的存取方式被後端用hardcode的方式導入`staticfiles`，這導致設置變成是多餘的部份，而且會跳過直接向django後端請求。
- 為符合前項的設置，iis被增加了一個靜態的link直接導向`staticfiles`。
- prefix有可能疊加多次，導致實際請求路徑與render的路徑會有出入。
- 使用middleware替換response的內容時，例如將部份字串從`static`->`django/static`，會導致header context length有變更而發生錯誤。

# 當前環境情況與解決方式
- 使用iis rewrite進行反向代理的設置，會重寫url，並導向至本機的8090 port。
- django專案需要透過以下幾個環境變數設置，讓相關的url在render的時候將prefix添加上：
    - `PROXY_PREFIX=/djangoai`：此設置代表所有用url變數解析時，
    - `FORCE_SCRIPT_NAME=/djangoai`：此設置代表存取靜態檔案實體的位置時要再加上prefix的部份
    - `PROXY_PREFIX_WRITE_SCRIPT_NAME=0`：此設置代表靜態檔案的路徑在產生的時候，不再增加prefix的部份
- django內建的static樣版函數會強制使用根目錄，無法與prefix的設置配合，解決方案為自訂標籤與樣版函數：
    - 自訂標籤置於：`webapps\common\templatetags`
    - 在開頭宣告：`{% load custom_tags %}`
    - 用 `{% custom_static 'portal/css/index.css' %}` 取代原先的 `{% static 'portal/css/index.css' %}`
- 因為本質上還是透過prefix在url添加字串，所以`.env`環境檔需要區分開發用與發佈用，而且直接在本機用8090 port訪問時，prefix仍然被添加，此時url的處理會是錯誤的。
